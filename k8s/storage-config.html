问题配置描述

可变配置 ​

ConfigMap 介绍 ​

主要管理容器运行所需的配置文件，环境变量，命令行参数等可变配置。用于解容器镜像和可变配置，从而保障工作负载(Pod)的可移植性。

yaml
apiVersion: v1
kind: ConfigMap
metadata:
  labels:
    app: flannel
    tier: node
  name: kube-flannel-cfg
  namespace: kube-system
data:
  cni-conf.json: |
    {
        "name": "cbr0",
        "type": "flannel",
        "delegate": {
            "isDefaultGateway": true
        }
    }
  net-conf.json: |
    {
        "Network": "172.27.0.0/16",
        "Backend": {
            "Type": "vxlan"
        }
    }
  • metadata 部分为 ConfigMap 元信息
  • data 部分管理2个配置文件

ConfigMap 创建 ​

创建命令

sh
kubectl create configmap [NAME] [DATA]
  • NAME ConfigMap 名称
  • DATA：
    • — 指定文件或者目录
    • — 指定键值对

指定文件 ​

sh
kubectl create configmap kube-flannel-cfg \
    --from-file=configure-pod-container/configmap/cni-conf.json \
    -n kube-system
yaml
apiVersion: v1
kind: ConfigMap
metadata:
  labels:
    app: flannel
    tier: node
  name: kube-flannel-cfg
  namespace: kube-system
data:
  cni-conf.json: |
    {
        "name": "cbr0",
        "type": "flannel",
        "delegate": {
            "isDefaultGateway": true
        }
    }
  • 文件名 cni-conf.json 最为 key
  • cni-conf.json 文件 json 内容作为 value

指定键值对 ​

sh
kubectl create confgimap special-config \
    --from-literal=special.how=very \
    --from-literal=special.type=charm
yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: special-config
  namespace: default
data:
  # 输入的键值对
  special.how: very
  special.type: charm

ConfigMap 使用 ​

ConfigMap 主要被 Pod 使用，一般用于挂载 Pod 用的配置文件，环境变量，命令行参数等。

配置环境变量 ​

yaml
apiVersion: v1
kind: Pod
metadata:
  name: cm-env-test
data:
  containers:
    - name: test-container
      image: k8s.gcr.io/busybox
      command: ["bin/sh", "-c", "env"]
      env:
        # 用 special-config 中的 special.how 定义环境变量
        - name: SPECIAL_LEVEL_KEY
          valueFrom:
            configMapKeyRef:
              name: special-config
              key: special.how
  restartPolicy: Never

配置命令行参数 ​

yaml
apiVersion: v1
kind: Pod
metadata:
  name: cm-cmd-test
data:
  containers:
    - name: test-container
      image: k8s.gcr.io/busybox
      # 在命令行参数中使用 ConfigMap 定义的环境变量
      command: ["bin/sh", "-c", "echo $(SPECIAL_LEVEL_KEY)"]
      env:
        - name: SPECIAL_LEVEL_KEY
          valueFrom:
            configMapKeyRef:
              name: special-config
              key: special.how
  restartPolicy: Never

挂载配置文件 ​

yaml
apiVersion: v1
kind: Pod
metadata:
  name: cm-volume-test
data:
  containers:
    - name: test-container
      image: k8s.gcr.io/busybox
      command: ["bin/sh", "-c", "ls /etc/config/"]
      volumeMounts:
        # ConfigMap 中指定的内容以文件形式挂载在容器中的 /etc/config 目录下
        - name: config-volume
          mountPath: /etc/config
  volumes:
    name: config-volume
    configMap:
      name: special-config
  restartPolicy: Never

ConfigMap 使用注意点 ​

  1. ConfigMap 文件大小限制: 1MB（*ETCD的要求）
  2. Pod 只能引用相同 Namespace 中的 ConfigMap
  3. Pod 引用的 ConfigMap 不存在时，Pod 无法创建成功。即 Pod 创建前需要先创建好 ConfigMap。
  4. 使用 envFrom 从 ConfigMap 来配置环境变量时，如果 ConfigMap 中的某些key被认为无效(比如key名称中带有数字)，该环境变量将不会注入容器，但是Pod可以正常创建。
  5. 只有通过 k8s api 创建的 pod 才能使用 ConfigMap ，其他方式创建的 pod (如 manifest 创建的 static pod )不能使用 ConfigMap

敏感信息 ​

Secret 介绍 ​

Secret 是在集群中用于存储密码、token等敏感信息用的资源对象。其中的敏感数据采用base-64编码保存，相比存储在明文的ConfigMap中更规范，更安全。

yaml
apiVersion: v1
kind: Secret
# Secret 元数据
metadata:
  name: my-secret
  namespace: kube-system
# Secret 类型
type: Opaque
data:
  # Secret 存储的护具
  username: YWRtaW4=
  password: MWYyZDFlMmU2N2Rm

Secret 主要类型 ​

  • type=Opaque
  • type=kubernetes.io/service-account-token
  • type=kubernetes.io/dockerconfigjson
  • type=bootstrap.kubernetes.io/token

Secret 创建 ​

Secret 创建可以是用户自己创建，也有 Secret 是系统自动创建

  • — 比如: K8S为每个namespace的默认用户(default ServiceAccount)创建的 Secret

手动创建命令:

sh
kubectl create secret generic [INAME] [DATA] [TYPE]
  • 其中DATA: 可以指定文件/键值对
  • 另外TYPE: 默认为Opaque

指定文件 ​

sh
kubectl create secret generic myregistrykey \
    --from-file=.dockerconfigjson=/root/.docker/config.json \
    --type=kubernetes.io/dockerconfigjson
yaml
apiVersion: v1
kind: Secret
type: kubernetes.io/dockerconfigjson
metadata:
  name: myregistrykey
  namespace: default
data:
  .dockerconfigjson: *****************

指定键值对 ​

sh
kubectl create secret generic prod-db-secret \
    --from-literal=username=produser \
    --from-literal=password=Y4nys7f11
yaml
apiVersion: v1
kind: Secret
type: Opaque
data:
  password: WTRueXM3ZjEx
  username: cHJvZHVzZXI=
metadata:
  name: prod-db-secret
  namespace: default

Secret 使用 ​

Secret 主要被Pod使用，一般通过volume挂载到指定容器目录，供容器中业务使用。另外在需要访问私有镜像仓库时，也可以引用Secret来实现。

挂载在用户指定目录下 ​

yaml
apiVersion: v1
kind: Pod
metadata:
  name: mypod
spec:
  containers:
    - name : mypodsecret
        image: redis
        # 以文件形式挂载在/etc/foo下
        volumeMounts:
        - name: foo
        mountPath: "/etc/foo"
        readOnly: true
  volumes :
    - name: foo 
      # 指定要挂载的 secret
      secret:
        secretName: mysecret

ServiceAccount 用的 secret ​

自动挂载在固定目录下(生成 ca.crt 和 token 两个文件)

yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-9f7575d75-mdfp6
  namespace: default
spec:
  containers:
    - image: nginx:1-alpine
      name: nginx
      # 固定排凯到容器中的/var/ruot日是下
      volumeMounts:
        - mountPath: /var/run/secrets/kubernetes.io/serviceaccount
          name: default-token-6fn7g
          readOnly: true
  # 使用 namespace 默认的 serviceAccount
  serviceAccount: default
  serviceAccountName: default
  volumes:
    - name: default-token-6fn7g
      secret:
        # 挂载 namespace 默认的 secret
        defaultMode: 420
        secretName: default-token-6fn7g

使用私有镜像库 ​

私有镜像仓库的信息可以通过Secret(kubernetes.io/dockerconfigjson)存储在集群中，Pod如果需要使用私有镜像仓库，可以通过如下两种方式来配置:

  1. Pod.spec.imagePullSecrets来指定secret
  2. ServiceAccount中设置imagePullSecrets，然后自动为使用该SA的Pod注入imagePullSecrets信息。
指定 imagePullSecret ​
yaml
apiVersion: v1
kind: Pod
metadata:
  name: private-reg
spec:
  containers:
    - name: private-reg-container
      iamge: <your-private-image>
  imagePullSecrets:
    - name: regcred
ServiceAccount 设置，Pod 创建时自动注入 imagePullSecrets ​
yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  creationTimestamp: 2015-08-07T22:02:39Z
  name: default
  namespace: default
  selflLink: /api/v1/namespaces/default/serviceaccounts/default
  uid: 052fb0f4-3d50-11e5-b066-42010af0d7b6
sercrets:
  - name: default-token-uudge
# 自动为使用该 ServiceAccount 的 Pod 注入 ImagePullSecrets 信息
imagePullSecrets:
  - name: myregistrykey

Secret使用的注意点 ​

  1. Secret文件大小限制: 1MB（*ETCD的要求）
  2. Secret虽然采用base-64编码，但是可以简单解码查看原始信息。因此机密信息采用Secret存储仍需要慎重考虑或者Secret访问者进行控制。对Secret加密有较强需求，可以考虑结合Kubernetes + Vault来解决敏感信息的加密和权限管理。
  3. Secret最佳实践: 因为list/watch的一般处理将获取到namespace下所有secret，因此不建议采取list/watch方式获取Secret信息。而推荐使用GET来获取需要的Secret，从而减少更多Secret暴露的可能性。

身份认证 ​

ServiceAccount 介绍 ​

ServiceAccount 主要用于解决 Pod 在集群中的身份认证问题。其中认证使用的授权信息，则利用前面讲到Secret(type=kubernetes.io/service-account-token)进行管理

SeviceAccount定义 ​

yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  creationTimestamp: 2019-06-03T08:32:59Z
  name: default
  namespace: default
  resourceVersion: "419220122"
  selfLink: /api/v1/namespaces/default/serviceaccounts/default
  uid: 320061cb-85da-1e9-909b-06b1d3b258c8
secrets:
  # 指定后台的Secret
  - name: default-token-6fn7g

ServiceAccount 对应的 Secret ​

*Secret 由 Serwiceaccount Controller 自动创建

yaml
piVersion: vl
data:
  namespace: ZGVmYXVsdA==
  # 用于校验服务端的证书信息，base-64编码
  ca.crt: LSOtLS1...
  # Pod 的身份认证用的 Token， base-64 编码
  token: ZXIKaGJHY2...
kind: Secret
netadata:
  # 关联的 ServiceAccount 信息
  annotations:
    kubernetes.io/service-account.name: default
    kubernetes.io/service-account.uid: 320061cb-85da-11e9-909b-06b1d3b258c8
  # -------------------------
  creationTimestamp: 2019-06-03T08:32:59Z
  name: default-token-6fn7g
  namespace: default
  resourceVersion: "419220120"
  selfLink: /api/v1/namespaces/default/secrets/default-token-6fn7o
  uid: 320182a6-85da-11e9-909b-06b1d3b258c8
# Secret类型
type: kubernetes.io/service-account-token
